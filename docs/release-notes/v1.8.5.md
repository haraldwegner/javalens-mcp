# javalens v1.8.5 — HTTP/SSE as the default transport

Theme: **stdio → HTTP/SSE as the default**. One long-running JVM per workspace can now serve N MCP clients over `POST /mcp` + `GET /mcp/events` instead of every client spawning its own private stdio child. Per-workspace tool count unchanged: **75**.

This is the foundation for the manager-side bug #9 fix (the 30 GB JVM leak when several Cursor + Claude windows are open): manager v0.15.0 hosts one resident fork JVM per workspace and emits URL endpoints in deployed MCP-client configs, replacing the stdio-spawn model that produces N × M JVMs.

## Strategic positioning vs upstream

Upstream Pzalutski-pixel/javalens-mcp at v1.3.5 is stdio-only. Multi-client shared service over HTTP/SSE is a capability gap stdio fundamentally cannot close — its "one JVM per client process" is structural, not a config knob. v1.8.5 establishes the fork as the only JDT-accurate MCP server that scales to N agents per workspace without N × JVM cost.

## What's new

### HTTP/SSE as the default transport

Launching `java -jar javalens.jar -data <workspace>` now starts an HTTP listener on `127.0.0.1:<ephemeral>` instead of reading JSON-RPC from stdin. The single READY line emitted to stdout carries the URL and token the manager (or any caller) uses to connect:

```
READY url=http://127.0.0.1:54321 token=<64-char-hex>
```

New CLI flags:

- `-transport stdio` — opt back to stdio for users who pin to v1.8.0 behaviour.
- `-port N` — bind to a specific port. Omit (or use 0) for OS-assigned ephemeral.
- `-bind 127.0.0.1` — bind address. Default `127.0.0.1`.
- `-token T` — supply a static Bearer token. Omit to generate a fresh 32-byte (64-char hex) token via `SecureRandom` at startup.
- `-token-file <path>` — record where to read the token from (reserved for future tooling).

The token MUST be presented as `Authorization: Bearer <token>` on every request; missing or wrong → 401. Localhost binding is the secondary security gate.

### POST `/mcp` (request-response)

JSON-RPC body in; JSON-RPC response out. Identical semantics to the stdio path — all 75 tools work unchanged because the protocol handler is transport-agnostic.

- `200 application/json` — successful dispatch.
- `204 No Content` — notification (no JSON-RPC id).
- `400` — malformed / empty JSON.
- `401` — missing or wrong Bearer.
- `405` — non-POST method.

### GET `/mcp/events` (Server-Sent Events)

Long-lived SSE stream of `event: <type>\ndata: <json>\n\n` frames:

- First frame on subscription: `event: ready\ndata: {}`
- `heartbeat` every 30 s (configurable internally) so manager-side health checks can confirm the JVM is alive end-to-end, not just holding the socket.
- The transport exposes a public `sendEvent(type, dataJson)` API for the tool layer to broadcast progress / partial-result events. **Tool-layer wiring is NOT in v1.8.5** — `McpProtocolHandler` remains pure request-response. The SSE channel ships heartbeats today; tools emitting progress events is a **v1.8.6 follow-up**.

## Why JDK `com.sun.net.httpserver.HttpServer`, not Jetty

The Eclipse target platform did not already include Jetty. Adding it would have meant a target bump plus the well-known risk of OSGi class-loader vs servlet-thread friction with JDT APIs. The JDK server avoids all of that: no target changes, no new bundle dependencies, no OSGi class-loader concerns, guaranteed with Java 21. SSE is straightforward chunked-response over the JDK server.

## Stdio still works

Stdio is supported indefinitely via `-transport stdio`. The protocol handler is transport-agnostic and the regression suite proves the stdio path is unchanged (StdioTransport JUnit + the full mcp.tests reactor, 561 tests, GREEN).

For users who upgrade the fork without upgrading the manager: existing v1.8.0 stdio-mode `args` configs will break (the default behaviour flipped). Either add `-transport stdio` to the args or upgrade to manager v0.15.0 which emits URL-based MCP-config entries.

## Migration notes

- **Existing manager users on v0.14.x**: upgrade to manager v0.15.0 as a coupled release pair. v0.15.0's deploy writer emits `{ url, headers: { Authorization: Bearer <token> } }` entries instead of `{ command, args }`, and the manager hosts one resident fork JVM per workspace.
- **Pure-CLI / non-manager users**: either keep launching with `-transport stdio -data <workspace>`, OR migrate to launching once + hitting `POST /mcp` from your tooling.
- **Antigravity / Cursor / Claude Desktop users**: configure your MCP entry with `{ "url": "http://127.0.0.1:<port>", "headers": { "Authorization": "Bearer <token>" } }`. The resident JVM survives across conversations, sidestepping the per-conversation orphaning bug reported on Google's Antigravity forum ([discuss.ai.google.dev #139866](https://discuss.ai.google.dev/t/bug-all-mcp-server-processes-are-orphaned-after-conversation-ends-accumulate-indefinitely-causing-system-memory-exhaustion/139866)) where stdio MCP children accumulate until RAM exhausts.

## What didn't change

- All 75 MCP tools work identically over HTTP and stdio. `tools/list` returns the same surface in both transports.
- `JavaLensApplication.start()` startup ordering and project auto-load (`workspace.json` lookup, `JAVA_PROJECT_PATH` fallback) unchanged.
- The McpProtocolHandler JSON-RPC dispatcher is unchanged.
