# javalens v1.8.6 — MCP protocol version negotiation

Hotfix on top of v1.8.5. **Fixes** the v1.8.5-shipped HTTP server hanging up modern MCP clients (Claude Code v2.1.x, etc.) at `initialize`.

## What was broken

`McpProtocolHandler.handleInitialize` hardcoded its response to the obsolete `protocolVersion: "2024-11-05"` (the original MCP spec from a year ago). Clients targeting newer specs send their preferred `protocolVersion` (e.g. `"2025-06-18"`) in the `initialize` request; per the MCP spec, the server must respond with a version the client supports or the client SHOULD disconnect.

End-user symptom: Claude Code's `/mcp` showed the connector as `failed` immediately on session start. No tool calls possible. Cursor's MCP integration was more tolerant and worked.

Verified by curl:

```
$ curl -sS -X POST http://127.0.0.1:8803/mcp \
    -H 'Authorization: Bearer ...' \
    -d '{"jsonrpc":"2.0","id":1,"method":"initialize", \
         "params":{"protocolVersion":"2025-06-18", \
                   "clientInfo":{"name":"claude-code","version":"2.1.168"}}}' \
    | grep protocolVersion
"protocolVersion":"2024-11-05"   ← bug: ignored the request
```

## What's fixed

`handleInitialize` now reads `params.protocolVersion` and echoes it back when it's in the server's supported set; otherwise falls back to `LATEST_PROTOCOL_VERSION`.

- `SUPPORTED_PROTOCOL_VERSIONS = { "2024-11-05", "2025-03-26", "2025-06-18" }`
- `LATEST_PROTOCOL_VERSION = "2025-06-18"`

Server logs the negotiation: `Protocol version: client requested 2025-06-18, negotiated 2025-06-18`.

The basic JSON-RPC + `tools/list` + `tools/call` surface is identical across these three spec versions, so version-echo is safe without adding new code paths. The newer-spec features (audio content, completion API, structured content) aren't advertised in `capabilities` and won't be invoked.

## Tests

- `initialize_echoesRequestedProtocolVersion_whenSupported`
- `initialize_fallsBackToLatest_whenRequestedUnknown`
- `initialize_fallsBackToLatest_whenAbsent`

Focused: `mvn -pl org.javalens.mcp.tests -am -Dtest='McpProtocolHandlerTest' verify` → 19 / 0 failures.

## What didn't change

- HTTP transport, Bearer auth, READY-line contract, SSE channel — all unchanged from v1.8.5.
- Tool surface unchanged: still 75 tools per server.
- Stdio fallback (`-transport stdio`) unchanged.
- No new CLI flags.

## Migration

Drop-in compatible with v1.8.5. The manager v0.15.0 auto-updates on next release-poller check; existing clients reconnect successfully.
