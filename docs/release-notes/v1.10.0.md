# javalens-mcp v1.10.0 — modernisation sweeps + Cursor DX + apply_cleanup/Lombok (Sprint 15)

Minor bump: **+2 tools (79 → 81)**, both parametric, plus two open bugs fixed and a
cross-cutting developer-experience pass. **This is the final javalens-branded release** —
the next sprint cuts over to the GOJA brand (goja-mcp v1.0).

## find_modernization — one parametric "what could this code adopt?" tool

Rather than six narrow `find_*` tools, modernisation ships as a single
`find_modernization(kind=…)` in the style of `find_quality_issue` /
`find_pattern_usages` — keeping the per-workspace tool budget low under the Antigravity
≈100-tool cap. Find-only (read-only); returns ranked `{ filePath, line, snippet, suggestion }`.

Eight kinds:

- `anon_to_lambda` — anonymous single-method implementations → lambda.
- `switch_to_pattern` — classic `switch` → switch expression / pattern switch (Java 21).
- `loop_to_stream` — accumulation for-each → Stream pipeline.
- `optional` — reference-returning methods that `return null` → `Optional<T>`.
- `class_to_record` — immutable data classes → `record` (the language-native answer to
  getter/setter boilerplate; `generate_getters_setters` now nudges records-first).
- `sealed` — abstract bases whose subclass set may be closed → `sealed` + `permits`.
- `lombok_to_record` — `@Data`/`@Value` data carriers → drop Lombok for a native record.
- `delombok` — any Lombok-annotated class → materialize members, remove the dependency.

## apply_cleanup — parametric clean-up catalog (upstream v1.4.2 harvest)

`apply_cleanup(kind=…)` applies safe, mechanical clean-ups via the v1.9.0 apply/undo
contract (`{ filesModified, diff, undoChangeId, summary }`; `auto_apply:false` to stage).
Deliberately **non-overlapping** with `organize_imports`, `format`, `apply_quick_fix` and
`find_modernization`:

- `add_final` — mark parameters and locals `final` when never reassigned. Reassignment is
  detected **by binding**, not name, so shadowing can't produce a false positive that
  breaks compilation.
- `redundant_modifiers` — strip modifiers implicit on interface members
  (`public`/`abstract` methods, `public`/`static`/`final` fields, `public`/`static`
  nested types).

`filePath` scopes to one file; omit it to sweep the whole project. A no-op returns
`hasChanges:false`.

## Lombok — removal *and* comprehension

Two orthogonal capabilities for Lombok-using code:

- **Removal** (the strategic, on-trend path — records replaced Lombok's main use):
  `find_modernization` `lombok_to_record` / `delombok`, backed by a source-only
  `LombokDetector` (import + annotation simple-name; needs no agent, no classpath).
- **Comprehension** (so analysis tools *understand* Lombok-using code): the product now
  bundles a pinned `lombok.jar` (1.18.36, version-locked to the bundled JDT 3.39 — the
  agent patches ECJ by shape, so it must match javalens's JDT, not the project's compile
  version). The **manager** detects Lombok in a workspace (build-file dep / `lombok.config`)
  and *conditionally* prepends `-javaagent:lombok.jar` to that resident JVM.
  **E2E-proven**: with the agent, `get_type_members` on a `@Data` class lists the
  synthesized `getX`/`getY`; without it (control), they're absent.

  > Comprehension activates only when **both** the fork (this release, ships `lombok.jar`)
  > **and** the manager (conditional `-javaagent`) are updated — a manager release carries
  > the injection side.

## Cursor developer-experience pass

From field feedback (`docs/javalens_feeback_from_cursor.md`):

- **FQN dot-form** — `find_references` & siblings now resolve `com.foo.Bar.method` /
  `.field` (the form agents naturally type), not only the `#` form. One `FqnResolver`
  change fixes all four `find_*` tools. (The `#`-form and private-member search already
  shipped in v1.8.0.)
- **analyze_method** — leads its response with the `resolvedSymbol` it actually resolved;
  a position miss returns nearby method candidates (with exact line/column) instead of a
  bare error; invalid-parameter errors carry an example arg shape.

## Bug fixes

- **#1 / bugs.md #15** — `change_method_signature` no longer mangles a constructor into an
  illegal method: it branches on `IMethod.isConstructor()` (never emits a return type) and
  rewrites constructor call sites by visiting `ClassInstanceCreation` / `this(...)` /
  `super(...)`, not only `MethodInvocation`.
- **#3 / bugs.md #14** — `health_check` reports the real manifest-derived version
  (`McpProtocolHandler.serverVersion()`) instead of a hardcoded `2.0.0-SNAPSHOT`.
