# javalens-mcp v1.9.0 — refactoring auto-apply + undo (Sprint 14b)

Minor bump: **every refactor tool's return shape changes** — the contract break this
version number signals. Pairs with no specific manager release; manager v0.15.1+ serves
it unchanged.

## The headline: refactor → verify → undo

Until v1.8.6 the fork's 10 local refactoring tools returned text-edit descriptions and
told the agent *"the caller should apply these edits"* — error-prone hand-application,
no revert story. The 5 structural tools auto-applied but discarded the undo. v1.9.0
ships one uniform contract across **all 21 mutating tools** (10 local refactorings,
5 LTK structural refactorings, 6 codegen tools):

- **Auto-apply by default.** One round-trip: the tool performs the change and returns
  `{ filesModified, diff, undoChangeId, summary }`.
- **Undo as a first-class handle.** `undo_refactoring(undoChangeId)` restores the
  pre-refactor state byte-for-byte. The agent's loop is
  `refactor → compile_workspace / run tests → keep, or undo`.
- **Opt-in staging.** `auto_apply: false` caches the built Change without touching disk
  and returns `{ changeId, diff, summary }`; `inspect_refactoring(changeId)` re-examines
  the diff, `apply_refactoring(changeId)` commits. For staging multiple refactors,
  inspecting a 40-file move before committing, or impact analysis.
- Staged changes and undo handles live in a session-scoped cache (1 h TTL, LRU-capped);
  apply and undo are one-shot.

New tools: `apply_refactoring`, `undo_refactoring`, `inspect_refactoring`,
`replace_duplicates` — **75 → 79 tools**.

## replace_duplicates — the find_duplicate_code loop closes

`find_duplicate_code` groups now carry a stable `groupId` (hash of the clone shape — no
session state, same id across calls). `replace_duplicates(cloneGroupId)` keeps a
canonical method and rewrites every **same-type** clone's body into a delegation call,
atomically in one Change with one undo handle. Cross-type clones are skipped with
per-instance reasons (cross-type delegation isn't automatically safe).

## readOnlyHint annotations — Cursor Ask mode unblocked

`tools/list` now marks the 64 detect tools (`find_*`, `get_*`, `analyze_*`, `search_*`,
`go_to_definition`, `health_check`, `list_projects`, `validate_syntax`,
`inspect_refactoring`, `suggest_imports`) with `annotations.readOnlyHint: true`
(MCP spec ≥ 2025-03-26). Restricted client modes — Cursor Ask mode field-reported the
block in June — can permit analysis without switching to an agent mode. Mutating tools
deliberately carry no annotations.

## Migration

- **Structural tools** (`move_class`, `pull_up`, …): behavior unchanged (they already
  applied); responses gain `diff` + `undoChangeId`, and the old `modifiedFiles`
  list-of-maps is now `filesModified` (list of path strings).
- **Local tools** (`rename_symbol`, `extract_method`, …): the big change — they now
  APPLY instead of returning `editsByFile`. Agents that hand-applied edits must stop;
  the refactor has already happened when the response arrives. Old behavior per call:
  `auto_apply: false`.
- **Codegen tools**: already applied; responses gain `diff` + `undoChangeId`.
- Pre-2025-03-26 MCP clients: the `annotations` field is additive and ignorable.
