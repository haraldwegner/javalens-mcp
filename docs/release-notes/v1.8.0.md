# javalens v1.8.0 ‚Äî lifecycle hygiene + agent-trust restoration

Theme: **lifecycle hygiene ‚Üí agent-trust restoration**. Nine real-world `docs/bugs.md` entries closed, three high-leverage tools added, dependency management extended to Gradle, fork docs scrubbed of proprietary product references. Per-workspace tool count: **73 ‚Üí 76**.

## Phase A ‚Äî bugs fixed (9)

### #1 (HIGH) ‚Äî `run_tests` Maven / Gradle / generic-Java launch path

`JUnitLaunchHelper.computeRuntimeClasspathMementos` builds an explicit pre-computed runtime classpath that bypasses the JDT JUnit launcher's `PluginClasspathProvider` for non-PDE projects. v1.7.1's `INVALID_PARAMETER`-with-workaround dispatch is removed; the tool now drives the real launch path. End-to-end happy-path JVM-spawning verification still waits on the fixture-build pipeline (`@Disabled` happy tests stay so); the precondition test (`runTests_plainMaven_doesNotNpeEndToEnd`) covers the NPE-on-`Bundle.getHeaders()` regression directly.

### #6 (MEDIUM) ‚Äî `WorkspaceFileWatcher` belt-and-braces robustness

Polled-with-timeout loop replaces the blocking `take()`; 50 ms debounce + event drain absorbs tmp+rename bursts; `OVERFLOW` events trigger unconditional reconcile; `try/catch` around the full loop iteration prevents silent thread death; a periodic mtime-fallback poll guarantees reconciliation every ‚â§ 2 s even if `WatchService` silently misses an event. The new `refresh_workspace` tool (Phase B.1) is the manual override if a watcher gap ever surfaces in production.

### #7 (LOW) ‚Äî `ProjectImporter` classpath dedupe

`addDependencyEntries()` now accumulates added lib + project paths in `Set<IPath>` and gates `entries.add(...)` on `Set.add()`. The fork's own multi-module Maven+PDE hybrid layout (and any similar Tycho hybrid where `lib/` + `.classpath`'s `<classpathentry kind="lib">` both contribute the same jar) loads without `setRawClasspath()` rejecting on duplicate entries.

### #8 (HIGH) ‚Äî `compile_workspace` `clean: bool` param

New `clean: bool` parameter (default `false`). When `true`, `CLEAN_BUILD` precedes `FULL_BUILD` so JDT discards its incremental cache. Required when a record's canonical constructor shape changes, a public method's signature shape changes, or any other edit the incremental builder skips recompiling consumers for. Default off preserves v1.7.x performance for the common case.

### #9 (HIGH) ‚Äî `compile_workspace` `scope: "main"|"test"|"both"` param

New `scope` parameter, **default `"both"`**. Diagnostics filtered by Maven `src/main/` vs `src/test/` path-segment convention; project-level errors always pass through. v1.7.x's silently-main-only behaviour was a bug ‚Äî `"0 errors"` on a signature change that broke test callers was the exact opposite of what "compile workspace" implies. **Behaviour change:** callers that relied on main-only diagnostics need `scope="main"`.

### #10 (MEDIUM-HIGH) ‚Äî `move_class` cross-project relocation + `modifiedFiles` populated

Sub-defects 1 + 2 fixed:

- **Cross-project relocation:** new `targetProjectKey` parameter (optional; falls back to auto-detect by matching the requested package against every loaded project's source roots). `sourceProject` derived from the CU's own `getJavaProject()` rather than the registry-supplied service. The file physically moves on disk; consumer imports update across both projects.
- **`modifiedFiles` populated:** `AbstractRefactoringTool.runRefactoring()` now registers an `IResourceChangeListener` (POST_CHANGE) around `PerformChangeOperation.run()`. Investigation showed JDT's `ProcessorBasedRefactoring$ProcessorChange` reports `children=0` ‚Äî its `perform()` does the work directly without exposing sub-changes ‚Äî which is why the previous tree-walking approach produced an empty array. The listener path catches them; the tree-walk is kept as a secondary path for refactorings that DO expose their tree (Rename etc.), deduped by formatted path.

**Sub-defect 3** (javadoc `@link` import filter) deferred to v1.8.1 ‚Äî intercepting `MoveCuUpdateCreator`'s internal `ImportRewrite` requires JDT internal-API spelunking beyond Sprint 14's bug-batch budget.

### #11 (MEDIUM) ‚Äî `PROJECT_KEY_DROPPED` distinct error code

New error code separates "you typo'd the key" (`INVALID_PARAMETER`) from "the workspace was rebalanced under you and your key is no longer live" (`PROJECT_KEY_DROPPED`). `JdtServiceImpl.removeProject()` records a 5-minute TTL'd drop marker; `IJdtService.wasRecentlyDropped(key)` exposes the timestamp. `AbstractTool.execute()` + every tool with its own `projectKey`-handling path (`compile_workspace`, `run_tests`, `move_class` `targetProjectKey`) check `wasRecentlyDropped` before falling through to `INVALID_PARAMETER`. Re-adding the same key clears the marker so a live project isn't shadowed. Callers can now branch on the code to re-acquire via `list_projects` instead of treating dropped-key as a typo.

### #12 (MEDIUM) ‚Äî `find_*` schema-honesty + graceful degradation + FQN-capability

All three halves shipped:

- **Schema-honesty.** All four `find_*` tool descriptions now spell out the position-based contract up front and document the FQN form (Phase B.2 below).
- **Graceful degradation.** `find_field_writes` no longer hard-refuses on near-miss positions ‚Äî non-field-at-position returns `SUCCESS` with empty `writeLocations` + up to 3 `nearbyFieldCandidates` (fields ¬±1 line of the requested position) so the agent can re-query.
- **FQN-capability** (also Phase B.2): see below.

### #13 (MEDIUM) ‚Äî `rename_symbol` constructor post-pass

When renaming a type, the new post-pass walks `IType.getMethods()`, identifies constructors via `IMethodBinding.isConstructor()` with a declaring-class key match, and rewrites any constructor identifier still bearing the OLD type name to the NEW type name. Idempotent on already-renamed code.

## Phase B ‚Äî new tools

### B.1 ‚Äî `refresh_workspace(projectKey?)`

Consolidated lifecycle tool that performs **all three** steps the v1.8.x backlog called for:

1. Refresh resources from disk (`IResource.refreshLocal(DEPTH_INFINITE)` per project).
2. Invalidate the JDT incremental compile cache (`CLEAN_BUILD` then `FULL_BUILD` per project).
3. Preserve `projectKey` state (does NOT drop or rotate keys).

Optional `projectKey` scopes to one project; absent ‚Üí refreshes every loaded project. Returns post-rebuild diagnostics (same shape as `compile_workspace`) plus `refreshedProjects: [...]` and a `summary` of files refreshed, class-files invalidated, error / warning counts.

### B.2 ‚Äî FQN overload for the `find_*` family

A new shared resolver (`FqnResolver` at `org.javalens.mcp.tools.fqn`) resolves type / method / field FQNs via `IJavaProject.findType` + erased-name parameter matching. Each of the four `find_*` tools now accepts the alternative `symbol` form alongside the existing `(filePath, line, column)` triple:

- `find_references(symbol="com.foo.Bar")` ‚Äî type
- `find_implementations(symbol="com.foo.Bar")` ‚Äî type
- `find_method_references(symbol="com.foo.Bar#methodName")` or `"com.foo.Bar#methodName(int,java.lang.String)"` ‚Äî method
- `find_field_writes(symbol="com.foo.Bar#fieldName")` ‚Äî field

Plus `scope: "workspace" | "project"` (with `projectKey?` when `project`). Method parameter matching accepts both fully-qualified and simple names ‚Äî `IMethod.getParameterTypes()` returns source-signature form (`QString;`) which `Signature.toString` renders as `"String"`; the matcher tries both.

Closes the coordinate-bisection cost for cross-project consumer mapping that was the most-repeated agent ask across every Sprint 1 / 3 / 4 / 6b session.

### B.3 ‚Äî `find_duplicate_code`

Workspace-scoped (or `projectKey`-scoped) clone detection. Strategy:

1. For every `IMethod` in scope, AST-extract the method body.
2. Tokenize via a portable regex tokenizer (the JDT `IScanner` is unreachable in Tycho's headless test runtime even with `Require-Bundle: org.eclipse.jdt.core` + `Import-Package: org.eclipse.jdt.core.compiler` present).
3. Normalize: identifiers ‚Üí `ID`, string literals ‚Üí `STR`, integer literals ‚Üí `INT`, float literals ‚Üí `FLT`, char literals ‚Üí `CHAR`, `null` ‚Üí `NULL`, `true`/`false` ‚Üí `BOOL`; keep operators, keywords, and punctuation.
4. Group methods by exact normalized sequence; emit `groups` of `instances: [{filePath, line, methodName, tokenCount, similarity, sourceProject}]`.

Knobs: `minTokens` (default 50), `threshold` (reserved for future fuzzy matching at 1.0), `crossProject` (default `false`). Read-only.

## Phase C ‚Äî Gradle path for the Ring 3 dep tools

`add_dependency`, `update_dependency`, `find_unused_dependencies` now detect Gradle as a fallback when `pom.xml` is absent and `build.gradle` / `build.gradle.kts` is present. Text-level Groovy DSL (`implementation 'g:a:v'`, both quote forms) + Kotlin DSL (`implementation("g:a:v")`) are handled by the shared `GradleBuildSupport` helper. Maven scope ‚Üí Gradle configuration mapping: `compile`‚Üí`implementation`, `test`‚Üí`testImplementation`, `provided`‚Üí`compileOnly`, `runtime`‚Üí`runtimeOnly`.

**Detection precedence:** `pom.xml` (Maven) takes precedence over `build.gradle*` (Gradle); hybrid projects with both pick Maven. Tool descriptions document this.

**Buildship target-platform integration deferred** to v1.8.x.x. The Gradle path's success response includes a warning hinting the caller to call `refresh_workspace` for classpath sync after a write ‚Äî the JDT-side classpath isn't updated synchronously.

## Compatibility

Drop-in upgrade from v1.7.1 with two behaviour deltas worth flagging:

- **`compile_workspace`'s default `scope` is now `"both"`** (was main-only). Test-source signature mismatches that v1.7.x silently passed now surface. Callers that relied on main-only diagnostics: pass `scope="main"`.
- **`find_field_writes` no longer fails on near-miss positions.** It returns `SUCCESS` with empty `writeLocations` and a `nearbyFieldCandidates` array instead of `INVALID_PARAMETER`. Callers that branched on the error response should branch on `writeLocations.size()`.
- **`PROJECT_KEY_DROPPED` is a new error code** distinct from `INVALID_PARAMETER`. Existing callers continue to see the `INVALID_PARAMETER` path for typos; long-lived sessions that previously couldn't recover from a workspace-state mutation now get a self-describing error pointing at `list_projects` re-acquisition.

Bundle-Version bumps `1.7.1.qualifier` ‚Üí `1.8.0.qualifier` across `org.javalens.core` / `org.javalens.core.tests` / `org.javalens.mcp` / `org.javalens.mcp.tests`. The reactor poms move from `1.7.1-SNAPSHOT` to `1.8.0-SNAPSHOT`.
